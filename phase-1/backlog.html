Backlog

The full list of deliverables, tasks, and work items scoped at the start of the project — organised by phase and maintained throughout as the team's reference for what was in and out of scope.

About the backlog

The project backlog was created during Phase One as part of the initial planning process. It translates the Statement of Work (SoW) into a concrete list of deliverables and tasks, giving the team a shared reference point for what the project was committed to producing.

The backlog lives alongside the Trello project management board — Trello handles individual task-level tracking within each heartbeat, while the backlog maintains the full-project view of what needed to be delivered across all phases.

Backlog vs Trello

The backlog is a high-level list of deliverables scoped at the start of the project. Trello is where individual tasks within each heartbeat are tracked day-to-day. The backlog is useful for assessing overall project scope; Trello is useful for knowing what's happening right now.

What the backlog covers

The backlog was structured around the phases agreed in the Statement of Work. Each phase had a set of defined deliverables. Below is the shape of the backlog by phase:

Phase One

Kickoff & Planning

Project charter, stakeholder mapping, user recruitment plan for research, visual project charter, and initial project management setup.

Phase Two

Discovery & Research

User research methodology, interview and focus group facilitation in collaboration with Torchbox, survey design and distribution.

Phase Three

Analysis & Findings

Analysis of qualitative and quantitative data, production of the Data Report: A little platform, a lot of strategy.

Phase Four

Community Strategy Design

Co-creation workshops, persona and user journey development, KPI and metrics framework, processes and workflows, recognition ecosystem design.

Phase Five

Technology Selection

Tech ecosystem mapping, platform comparison, user stories, demographic considerations, and the Technical Discovery Report and Recommendations.

Pilot Prep & Post-Pilot

Pilot Preparation & Post-Pilot Resources

Platform configuration, content creation, user testing, accessibility review, and post-pilot planning documents including the readiness framework and integration roadmap.

How the backlog was used

The backlog served as a check against scope creep and a reference for acceptance criteria — helping the team and AIUK agree on what "done" looked like for each deliverable. It was also used at retrospectives to assess overall project progress.

Acceptance criteria

Each deliverable in the backlog had agreed acceptance criteria — the conditions that needed to be met before it could be marked as complete. These were negotiated between the WAO and AIUK teams, as described in the Project Charter.

The backlog is an internal project document. The full document index — which lists all completed deliverables — is the public-facing companion to the backlog, showing what was actually produced across the life of the project.
