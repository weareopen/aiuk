Accessibility
Testing

A basic accessibility audit of the Amnesty Community platform — covering automated checks, manual review, screen reader and keyboard navigation testing, and theme colour contrast improvements — confirming the platform is in good shape ahead of the pilot.

Introduction

Accessibility is fundamental to creating an inclusive digital experience for the Amnesty Community. It is both an ethical commitment and a practical requirement.

This accessibility check is intentionally basic in scope. We used the Accessibility tools built into the Firefox web browser along with the IBM Equal Access Toolkit. These both use established accessibility standards aligned with WCAG 2.2 (A / AA).

This document is held in Google Docs and is accessible to project team members.

Limitations of automated testing

There are things that cannot be caught by automated tools:

  1. Context and usability – automated tools cannot determine whether alt text is meaningful or whether the navigation flows logically (e.g. alt="picture1234.jpg" would pass automated tests but does not actually provide any useful information to screen reader users)
  2. Complex interactions – users with motor impairments might find dropdown menus, modal dialogues, and form behaviours problematic, but these would not be flagged by automated detection.
  3. Content clarity – community members using assistive technologies often resize content, and automated tools cannot assess whether this works properly without increasing line length – or whether error messages are clear.
88%+
Lowest score during basic testing Scores in the high 80s or above on a 0–100 scale are generally considered highly accessible. 90+ is considered strong and broadly compliant with all major web accessibility guidelines. Our lowest score during this basic testing was 88%, showing that we are already highly accessible.

Some tests returned outputs we are unable to influence. For example, some HTML elements require ARIA roles for full accessibility. Not only would this require changes in the Discourse software itself, ARIA roles can serve to degrade accessibility in complex software. There may be a good reason why the Discourse software does not implement ARIA roles. Such improvements may need to be dealt with programmatically, likely within the core software, or through custom development to come later.

During the Pilot, we will learn directly from community members with accessibility needs. Their real-world experience using the Amnesty Community will reveal gaps that automated tools cannot detect and will inform decisions about more comprehensive testing after the pilot concludes.

Amnesty can then invest in fully-featured accessibility services and tools based on evidence from this initial phase and user feedback.

Main issues found

The following sections detail each issue identified during testing, along with the action taken and the resulting outcome.

# Issue Action taken Result
1 Footer contrast too low Matched colour to primary grey (#1a1a1a) from updated Amnesty style guide (Torchbox) ✅ Pass (automated)
2 Logo missing text alternative Confirmed the Amnesty logo has a text alternative for screen readers; error resolved ✅ Pass (manual)
3 Links opening in new tab/window without notification Modified link targets so that links do not open in a new tab or window ✅ Pass (automated)
4 Search box contrast too low Improved icon contrast so users can see where to click; the 'Search' word colour could not be changed ✅ Pass (manual)
5 Contrast on column headers and other elements too low Improved contrast by darkening the grey ✅ Pass (automated)

Issue 1: Footer

The contrast of the footer was too low. On reviewing the Amnesty style guide created by Torchbox, we noted that it has received an update, so we matched the colour to primary grey (#1a1a1a).

Result

✅ Passes automated accessibility check

Issue 2: Logo

We discovered that the Amnesty logo does, in fact, have a text alternative for screen readers, therefore this error is fixed.

Result

✅ Passes manual accessibility check

Issue 3: Links

Links should either not open in a new tab or window, or the user should be visually notified that this should happen. We have modified targets so that links do not open in a new tab or window.

Result

✅ Passes automated accessibility check

Issue 4: Search box

The contrast of the search box was too low. Although we have not been able to change the colour of the 'Search' word, we have improved the icon contrast so that users can see where to click.

Result

✅ Passes manual accessibility check

Issue 5: Contrast on various elements

The contrast of column headers and other elements was too low, so we improved the contrast by darkening the grey.

Result

✅ Passes automated accessibility check

Tabbing order

The tabbing order test shows the order in which users pressing 'Tab' on a keyboard will be able to access elements. Testing confirmed that the hierarchy is working appropriately — keyboard navigation moves through the page in a logical sequence that reflects the visual structure.

Themes

Over the past months, we have applied Torchbox design elements to the AIUK Discourse install by creating a custom theme, with custom styling, and two colour palettes (a light palette and a dark one). During accessibility testing, we found that calculated colour values did not provide enough contrast. We asked our hosting partner Communiteq to add more colour options to the Admin interface, so that we would have more nuanced control over the displayed colours.

This allowed us to quickly and easily address multiple accessibility issues.

Light theme

Light theme contrast issue

The contrast of grey elements on the light theme was 3.95:1, which is slightly below the 4.5:1 required to meet WCAG standards. We have fixed this using Amnesty's colour palette and choosing greys with more contrast.

Contrast loss simulation

The "contrast loss" simulation shows what the platform might look like for a user with low vision. This simulation was used to inform our colour adjustments and ensure improvements would be meaningful for users with visual impairments.

Dark theme

Dark theme contrast issue

The dark theme had a lower contrast ratio of 2.73:1, which is below WCAG requirements. We have adjusted colours to bring both themes into compliance.

Conclusion

We feel that this basic accessibility testing showed that we are in good shape for the initial pilot. We have addressed a number of issues that would have been difficult to find without testing, and we have pointed out some areas for future improvements. As mentioned above, accessibility testing is an area that AIUK may wish to further invest in as the digital transformation project moves forward.

Summary of outcomes

  • All five identified issues have been resolved — each passed either automated or manual accessibility checks after remediation
  • Lowest automated score was 88% — above the threshold generally considered highly accessible
  • Screen reader and keyboard navigation confirmed working — tabbing order follows a logical hierarchy
  • Both light and dark themes improved — colour contrast updated to meet WCAG 4.5:1 requirements
  • Future investment recommended — comprehensive accessibility services and tools should be considered based on evidence from the pilot and user feedback
